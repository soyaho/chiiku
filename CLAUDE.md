# 開発規約（Claude向け）

**claude-dev-template から立ち上げたプロジェクトは、この規約に従って開発する。**
このテンプレートは対象領域を規定しない——プロジェクト固有の規約は、下の
「このプロジェクトの構造」「検証コマンド」「プロジェクト固有の約束事」を
育てることでこのファイルに集約される。

**プロジェクト: そうぞう（chiiku）** — 6歳児（年長）向け知育ゲーム集。
製品名は「そうぞう」（想像/創造＋音の中の「ゾウ」。タイトル画面のゾウがマスコット。v0.6）。
コアの価値は「動詞が先・世界が答える・✗を出さない」という設計原則がデジタルで成立するかの検証。
- **てんびん**: 数量の演算に世界律で応答する自由遊び（皿に載せる＝足し算、動物に食べさせる＝引き算、
  天秤の傾き＝比較）。子どもが自分の目標（同じにする・食べさせる）を持ち込め、世界がそれに答える。
  v0.3 までの5問ドリルと旧第2ゲーム「わけっこ」は v0.4 で自由遊びに統合・引退
  （経緯は KNOWLEDGE.md 棚卸し記録）
- **おふね**: 分割と反復（あまりのあるわり算の身体感覚）。世界律は「川の流れ」1本——
  積むと喫水が下がり、積みすぎた分は水があふれて流れ戻る（容量は教えず物理で発見させる）。
  舟は押し出しで出航し、空になれば流れが戻す。仕様: `docs/ofune_spec_v0.1.md`
- 仕様書: `docs/tenbin_spec_v0.1.md`（基底）＋差分 v0.2（統一落下物理）・v0.3（画面遷移）・
  v0.4（自由遊び化・数量演算ゲート）、`docs/ofune_spec_v0.1.md`（おふね）、
  `docs/sozo_spec_v0.6.md`（そうぞうブランド・置き場所保持吸着・舟の静止捕捉/held/甲板可逆）、
  `docs/sozo_spec_v0.7.md`（おおもの＝重さ2の果物・投げの誠実さ。傾き/容量は重さ合計基準）

## このプロジェクトの構造

設計・実装・監査の全員がここを前提知識として読む。構造が変わったら必ず更新する：

- **成果物**: `index.html` 1ファイルのみ（外部依存・アセット・ビルドなし）。Canvas 2D＋WebAudio 合成音。
  論理座標 1024×768 固定、iPad Safari（横持ち・PWA）想定。コードはセクションコメントで区切る
  （後続ミニゲームへの部品昇格を見据える。v0.1 では本格分離しない）
- **状態・データ**: 永続＝`localStorage` キー `tenbin_logs`（v2・ラウンド制）・`ofune_logs`（v1）・
  `wakekko_logs`（legacy・読み取りとコピーのみ）・`tenbin_prefs`（設定 JSON）。
  旧形式・破損データも falsy 安全＋数値正規化で扱いログ喪失させない。
  揮発＝画面遷移（title/menu/game）・ラウンド進行・物理・ドラッグ状態（保存しない。リロードでタイトルから）
- **実インターフェース**: canvas へのタッチ（touchstart/move/end）とマウスフォールバック。
  画面遷移: タイトル→メニュー→ゲーム、ゲームからは右上おうちボタン0.8秒押し込みで戻る。
  検証はこの面（実イベント送出）を通す。読み取り専用フック `window.__tenbin` と
  テスト入力注入 `?seed` / `?timescale` / `?game` の契約は `docs/test-hooks.md`
- **クリティカル欠陥の定義**（このプロジェクトで絶対に出さない欠陥クラス）:
  1. **進行不能** — 果物が操作不能・喪失（未知状態・総数減）になる・ラウンド完了が発火しない・
     舟が水中で停止したまま戻らない・画面遷移（タイトル/メニュー/ゲーム/おうち/幕間）で行き止まりになる
  2. **禁止演出の出現** — ✗印・ブザー・失敗アニメ・減点・制限時間・激しい点滅・
     動物の拒食（食べてくれない）、および**ゲームプレイ画面内**での文字・数字の描画
     （タイトル・メニュー・親メニューは文字可）
  3. **ログの喪失・破損** — `tenbin_logs` / `ofune_logs` / legacy `wakekko_logs` が壊れる・消える・
     旧データや破損データで起動不能

## 検証コマンド

```bash
node harness/run.js
```

Playwright（グローバル導入済み。npm install 不要）＋事前配置 Chromium で、index.html を
実イベント（マウス＝タッチのフォールバック経路）で操作する E2E 一式。
1セッション完走・ドロップ/戻しのログ・釣り合い演出・向き検出・文字禁止ガード・親メニューまで通す。

## ディレクトリ構成

- `index.html` — 成果物（ゲーム本体、1ファイル）
- `docs/tenbin_spec_v0.1.md` — 仕様書（正）
- `docs/test-hooks.md` — テストフック契約（`window.__tenbin` / `?seed` / `?timescale`）
- `harness/run.js` — E2E 検証ハーネス（検証コマンドの実体）
- `KNOWLEDGE.md` — 設計原則（持ち越し資産）＋バグ台帳。変更前に必ず参照する
- `.claude/` — 開発ワークフロー一式（下記）

## `.claude/` の構成（この規約を動かす仕組み）

- **スキル** `.claude/skills/`（`/コマンド` で起動）
  - `/change` — 成果物に触る変更の必須パイプライン（下記の要約）
  - `/feedback-fix` — ユーザー／テスターの指摘を原因診断→修正→緑→台帳まで一気通貫で処理
  - `/run-checks` — 検証コマンドの実行
  - `/backflow` — ワークフロー・一般化した設計原則をテンプレート正本へ還流
- **サブエージェント** `.claude/agents/`（Agent ツールで起動）
  - `implementer` — 実装担当（Sonnet）。まとまったコード変更はこれに委任する
  - `advisor` — 設計判断・行き詰まりの相談役（上位モデル・読み取り専用）
  - `auditor` — クリティカル欠陥の独立監査（読み取り専用）
  - `verifier` — 検証コマンドの実行と結果要約
- **台帳** `KNOWLEDGE.md` — 設計原則＋バグ台帳（症状→根本原因→対策）。
  **原則0＝測れないものは設計工程で守る、が全てに優先。**
  検証の緑は正しさの証明ではない——測れないもの（コアの価値・体験）は設計工程で守る
- **メモリ** `.claude/memory/` — セッションをまたぐ記憶（下記）

## メモリ `.claude/memory/`（セッションをまたぐ記憶）

環境の癖・決定の経緯・ユーザーの流儀など、**次のセッションの自分に引き継ぐ文脈**を保存する。
**1事実 = 1ファイル**の Markdown。組み込みメモリではなく必ずここに保存する
（リポジトリと一緒に移動できるようにするため）。索引は `INDEX.md` ——
**セッション開始時に目を通し**、必要なファイルだけ開く。

- **書くもの**: 環境の癖（権限・承認・外部ツールの制約）、決定の経緯（なぜこの設計を
  選び何を却下したか）、ユーザーの流儀（好み・確認の要否・コミュニケーションの型）
- **書かないもの**: 規約になるもの（→ CLAUDE.md）、原則・バグ（→ `KNOWLEDGE.md`）。
  メモリは「他に置き場のない文脈」の受け皿——恒久ルール化できた時点で然るべき場所へ昇格させ、
  メモリ側は消す
- 追加・更新したら `INDEX.md` の一覧も同じコミットで更新する

## 変更パイプライン（必須。詳細は `/change` スキル）

成果物に触る変更は、必ずこの順で行うこと：

1. **設計** — 変更内容と影響範囲を、「このプロジェクトの構造」の語彙（状態・インターフェース・
   クリティカル欠陥の定義）で列挙。`KNOWLEDGE.md` の設計原則に照らす。
   **特に原則0**：その変更はコアの価値に向かっているか、摩擦の解消がコアの価値を削っていないか
2. **実装** — 規模が大きい場合は `implementer` サブエージェントに委任し、本文脈は設計とレビューに回る。
   委任時は仕様・対象ファイル・完了条件をプロンプトに全部書く（サブエージェントはこの会話を見られない）。
   **同じ問題で実装が2回失敗したとき・設計上の分岐で確信が持てないときは `advisor` に相談する**
   （状況・試したこと・関連ファイルパスを添える）。些細な判断（命名・等価な実装の選択）は相談せず自分で決める
3. **検証（緑必須）** — 上の「検証コマンド」を実行して全て green。
   新しい振る舞い・進行を足したら**同じコミットで**検証（テスト・シナリオ）を拡張する
4. **独立監査** — 「クリティカル欠陥の定義」に関わる箇所（永続データ・状態遷移・公開
   インターフェース）に触れたら、`auditor` エージェントを別文脈で回す。
   所見は**検証で再現できたものだけ**採用
5. **ノウハウ反映** — 新種のバグは `KNOWLEDGE.md` に追記し、可能なら自動検証（テスト・invariant）に
   落とす。**台帳に書くだけで終わらせない**
6. **還流** — この変更で `.claude/` のワークフローを改良した、または設計原則に一般化できる学びを
   足した場合は、**同じセッションのうちに `/backflow`** を実行してテンプレート正本へ還流させる

## 検証の絶対原則

- **実インターフェース主義**: 検証は成果物の**実インターフェース**（実際の UI 操作・CLI 呼び出し・
  公開 API）を通して行う。内部関数の直呼び・内部状態の直書きで「動いた」ことにしない
  （内部状態の**読み取り**はアサーション用に可）。
  - 理由: 内部を呼ぶテストは「利用者が実際にその結果に到達できるか」を証明しない
  - 例外は**テスト入力の注入**（保存データ・乱数シード・設定ファイル等）のみ。
    これは「入力データ」であり、内部状態の直接書き換えとは区別する
- 通しシナリオが最後まで通ること自体が「主要な流れが実インターフェースで到達可能」の証明
- **緑＝正しさではない**: 検証は到達可能性・回帰を測るが、コアの価値・使いやすさは測れない。
  測れないものは検証ではなく設計工程で守る（原則0）

## 実装上の約束事

### どの開発でも守ること（普遍）

- 永続データ（セーブ・DB・設定）は分離して設計し、新しいフィールドは旧データに
  無くても安全（falsy 安全・デフォルト補完）にする
- 重要な状態の確定・保存は**その原因となる操作の瞬間**に行う（遅延タイマー・非同期
  コールバックの中で確定させない）。やむを得ない場合は読み込み時の逆算修復を追加し、
  復旧の検証ケースを足す
- 遅延実行される処理は、発火時に前提（状態・画面・文脈）がまだ成立しているか再検証してから実行する
- 新しい振る舞いを足したら**同じコミットで**検証を拡張する。構造・約束事が変わったら
  CLAUDE.md（「このプロジェクトの構造」「固有の約束事」）も同じコミットで更新する
- `.claude/` のワークフローを改良したら同じセッションで `/backflow`
- コミットメッセージには検証結果（pass/fail の要約）を一行入れる

### このプロジェクト固有の約束事（立ち上げ時から育てる）

- **文字・数字はゲームプレイ画面内で描かない**（`fillText`/`strokeText` は
  タイトル・メニュー・親メニューのみ。ハーネスがラップして違反を検出する）。
  プレイ中の数は進行ドット等の図形で表す。絵文字は全画面で使わない
- **数量演算ゲート**: 世界の新しい応答は数量の演算（載せる＝足し算、食べさせる＝引き算、
  傾き＝比較）であるものだけを許す。装飾的応答は追加しない（仕様 v0.4 §1）
- **✗・ブザー・失敗演出・制限時間・激しい点滅（1秒3回以上）・動物の拒食を実装しない**。
  「失敗フィードバック」を要求されたら、原則0に照らして設計から練り直す
- ゲームループは**固定タイムステップ（dt=1/120s）のアキュムレータ方式**。ゲーム内の全タイマー
  （静止判定・遷移・長押し・演出・ログms計測）はシミュレーション時間基準にする。
  `?timescale` は積分回数を増やすだけで dt は変えない（物理の安定性を守る）
- 全乱数はシード可能な RNG（`?seed`）を経由させる。`Math.random()` を直接呼ばない
  （例外はシード未指定時のシード源のみ）。**演出専用の乱数（BGM・効果音ノイズ等）は
  ゲーム進行の rng を消費させず、別の固定シード rng を使う**（配置の決定性を守る）
- `window.__tenbin` は **getter のみ・読み取り専用**を保つ。テスト用の書き込みフックを足さない
  （テスト入力の注入は `?seed` / `?timescale` / `?game` の3つに限る。契約: `docs/test-hooks.md`）
- 天秤のバネ-ダンパは**臨界減衰にしない**（1〜2回ゆらっと揺れる触感がコアの価値。
  ハーネスの spring テストがオーバーシュート回数を検証する）
