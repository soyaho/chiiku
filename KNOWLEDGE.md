# ノウハウ台帳

このファイルは「設計原則（持ち越し資産）」と「見つけたバグのクラス → 恒久対策」を
蓄積する台帳。**新種のバグを見つけたら必ずここに追記し、可能な限り自動検証
（テスト・invariant・シナリオのアサーション）に落とすこと。台帳に書くだけで終わらせない。**

下の設計原則は、前作群（ゲーム開発の系譜: proto_ae → html-tap-adventure-template →
html-game-template）で実際に踏んだバグ・ユーザー指摘から、**開発一般に通じる形に
一般化して**持ち越したもの。領域固有の原則（ゲームUI・遊びのデザイン等）は各ドメイン
テンプレート側の台帳にあり、ここには持ち込まない。**このプロジェクトで新しく学んだ
ことは、原則なら「設計原則」へ、個別バグなら「バグ台帳」へ足す。**

## 設計原則（開発一般）

### 価値の原則（全原則に優先）

0. **測れないものは設計工程で守る（コア価値ゲート）** — 検証の緑は正しさの証明ではない。
   到達可能性・回帰・遮蔽は測れるが、**コアの価値（そのプロダクトが与える体験・解く問題）は
   測れない**。測れるものだけが最適化される——だからコアの価値は検証ではなく設計工程で守る。
   - 機能を足す・直す**設計段階**で必ず問う：「この変更はコアの価値に向かっているか？
     汎用の道具・ありふれた実装に差し替えても同じなら、コアの語彙で言い換えられないか？」
   - **ユーザー／テスターは摩擦（詰まる・分かりにくい・遅い）は報告できるが、
     価値の欠落は報告できない。** 摩擦の指摘だけを食べ続けると、全ての改善が
     「無難な道具としての完成度」に向かう（前作の実例: 30回以上の改善が全部そうだった）。
     フィードバック対応のたびに「コアの価値の語彙でこの問題を解けないか」を一度通す
   - 定期的に（数リビジョンごと）主要機能を棚卸しして、コアの価値に照らすレビューを行い、
     結果を台帳に記録する

### 検証の原則

1. **実インターフェース主義** — 検証は成果物の実インターフェース（実際の UI 操作・CLI 呼び出し・
   公開 API）だけを通して行う。内部関数の直呼び・内部状態の直書きで「動いた」ことにしない
   （内部状態の**読み取り**はアサーション用に可）。内部を呼ぶテストは全緑でも
   「利用者がその結果に到達できるか」を何も証明しない
   （前作の実例: 入力の横取りによる進行不能を、内部関数直呼びのテストが長期間素通しした）。
   例外は**テスト入力の注入**（保存データ・乱数シード・設定ファイル等）のみ——
   「入力データ」であり、内部状態の直接書き換えとは区別する
2. **通しシナリオ＝到達可能性の証明** — 主要な流れ（ユーザーが価値に到達する経路）を
   実インターフェースで最初から最後まで通すシナリオを常備する。
   振る舞い・進行を追加・変更したら、そこに実操作で到達するステップを同じコミットで足す
3. **非同期の安定待ち** — 遅延発火・アニメーション・非同期処理があるなら、テストは
   「安定状態が一定時間続く」まで待ってから次の操作をする（固定 sleep はフレークの温床）
4. **実行環境マトリクス** — 利用者が実際に使う環境（画面サイズ・OS・ランタイムのバージョン等）
   を列挙し、検証は全環境で回す。環境依存の破れは該当環境でしか検出できない
   （前作の実例: 縦持ちだけ直して横持ちの表示崩れを見逃した）
5. **到達可能性≠発見可能性** — 正解を知っているテストは「利用者が答えに気づけるか・
   使い方が分かるか」を検証しない。UX・判読性・ドキュメントの分かりやすさは、
   実寸・実環境での目視／実使用レビューで別途確認する

### 状態・データの原則

6. **重要な状態は「操作の瞬間に確定」か「読み込み時に逆算修復」** — 状態の確定・保存を
   遅延タイマーや演出・非同期のコールバック内に置くと、その窓で中断されたとき
   中途半端な永続データが焼き付く（前作の実例: 確定が演出コールバック内にあり、
   詰む中間セーブ窓が5箇所あった）。原因となる操作の瞬間に同期的に確定するか、
   やむを得なければ読み込み時の逆算修復（reconcile）を必ず対で実装し、復旧の検証ケースを足す。
   修復では「導出できる値（再計算してよい）」と「操作で確定した値（補完のみ・取り消し禁止）」を区別する
7. **入力・操作の対象を曖昧にしない** — 利用者の1操作が何に作用するかを常に一意にする
   （UI ならクリック対象の重なり・遮蔽、CLI なら曖昧なフラグ・破壊的操作の暗黙対象、
   API なら同名の多義エンドポイント）。曖昧さは構造で断ち、可能なら自動検証（invariant）で恒久検出する
8. **遅延実行は発火時に前提を再検証する** — setTimeout・キュー・ジョブなど遅延して走る処理は、
   発火時に前提（状態・画面・文脈）がまだ成立しているかガードしてから実行する
   （利用者は待ってくれない——その間に状態は変わる）。状態依存の決定は1関数に一元化し、
   通常経路と復帰経路の双方で同じ関数を使う

### インターフェースの原則

9. **利用者は製品世界のルールで操作を予測する——世界律の特例を作らない** — 製品が世界を
   物理（あるいは一貫した比喩・文法）で提示するなら、操作の結果もその律で解決する。
   実装都合のショートカット（近接スナップ・暗黙のオートコレクト・特定座標だけ効く操作等）は
   利用者の自然な予測を裏切り、「正しい操作をしたのに何も起きない」という摩擦報告として現れる
   （本作の実例: 皿の真上から落としても載らなかった——子どもは世界を物理として信じていた。
   リリース＝常に落下体・皿が捕捉、という統一物理で解決）。この種の報告を受けたら
   個別の判定緩和ではなく**世界律の一貫性の破れ**を疑う

## 追記ルール

- 1バグ = 1行。**症状 → 根本原因 → 対策** をこの順で。対策は「どのテスト／invariant に実装したか」まで書く
- 修正だけで済ませず、「同じクラスのバグが再発したら検証が検出できるか？」を自問する。
  できないなら検証を足す
- 原則として一般化できる学びは「設計原則」の節にも番号を振って足す（→ 足したら `/backflow` で正本へ）
- **肥大化したらアーカイブ**（ゲーム系譜の台帳運用から一般化）: バグ台帳が30行を超えたら
  （または全体が300行を超えたら）、対策がコード化済み（テスト／invariant に実装済み）の
  古い行から順に `KNOWLEDGE-archive.md` へ移す。本体には「#番号｜一行要約」の索引を残す。
  設計原則は生きた知識なので、原則はアーカイブせず本体に残し続ける

## サブエージェント運用の雛形

クリティカル欠陥の定義（CLAUDE.md）に関わる変更では、実装とは別の文脈で監査を回す：

- **独立監査**（auditor）: 「CLAUDE.md のクリティカル欠陥の定義を監査観点に、該当する経路を
  逆向きに辿って成立条件を検証。遅延実行×状態遷移の競合、中断・復帰で壊れる永続データ、
  入力対象の曖昧さを重点確認。コード引用で根拠を示す。編集禁止」
- **検証実行**（verifier）: 検証コマンドを実行し、fail/warn を要約させる
- 監査所見は鵜呑みにせず、検証で再現できたものだけをバグとして扱う
  （再現できたら台帳へ、テスト／invariant へ反映）

## 既知の制約

- **緑＝正しさではない。** 検証は到達可能性・回帰を測るが、コアの価値・使いやすさは測れない。
  測れないものは検証ではなく設計工程（原則0）で守る
- このテンプレートは検証の実体を持たない。検証コマンドが未整備の間は、変更パイプラインの
  第3工程が空洞になる——**立ち上げ直後に検証の器を作ることが最優先タスク**
