# 変更パイプライン

成果物に触る変更は、必ずこの順で行う。**一つでも飛ばさない。**
前提知識として CLAUDE.md の「このプロジェクトの構造」（成果物・状態/データ・
実インターフェース・クリティカル欠陥の定義）を先に読む。

## 1. 設計
- **最初に原則0（コア価値ゲート）**：この変更はコアの価値に向かっているか、
  摩擦の解消がコアの価値を削っていないかを問い、判定を設計メモに書く。
- 変更内容と影響範囲を、「このプロジェクトの構造」の語彙（永続データ・状態遷移・
  実インターフェース・クリティカル欠陥の定義）で列挙する。
- 該当する設計原則（`KNOWLEDGE.md` の「設計原則」）に照らす。

## 2. 実装
- 規模が大きければ `implementer` サブエージェントに実装を委任し、本文脈はレビューに回る。
- 約束事（`CLAUDE.md`「実装上の約束事」——普遍の節と、プロジェクト固有の節の両方）を守る。
  特に：新しいフィールドは旧データで falsy 安全に、重要な状態は操作の瞬間に確定、
  入力・操作の対象を曖昧にしない。
- **同じ問題で実装が2回失敗したとき・設計上の分岐で確信が持てないときは `advisor` に相談する。**

## 3. 検証（緑必須）
- CLAUDE.md「検証コマンド」を実行し、全て green であること。
- **新しい振る舞い・進行を足したら、同じコミットで検証（テスト・シナリオ）を拡張する**
  （その振る舞いに実インターフェース経由で到達するステップを足す）。
- 実インターフェース主義：内部関数の直呼び・内部状態の直書きで「動いた」ことにしない
  （読み取りのみ可。例外はテスト入力の注入のみ）。
- 検証コマンドが未整備なら、この変更より先に検証の器を作る。

## 4. 独立監査
- 「クリティカル欠陥の定義」（CLAUDE.md）に関わる箇所——永続データ・状態遷移・
  公開インターフェース——に触れたら、**`auditor` エージェント**を別文脈で回す。
- 所見は鵜呑みにせず、**検証で再現できたものだけ**をバグとして扱う。

## 5. ノウハウ反映
- 新種のバグを見つけたら `KNOWLEDGE.md` のバグ台帳に1行追記（症状→根本原因→対策）。
  **書くだけで終わらせず**、可能なら自動検証（テスト・invariant）に落とす。
- プロジェクトの構造・約束事が変わったら、CLAUDE.md の該当節も同じコミットで更新する。

## コミット
- コミットメッセージに**検証結果（pass/fail の要約）を一行**入れる。
- 指定された作業ブランチにコミット・プッシュ。
