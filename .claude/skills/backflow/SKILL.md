# ワークフロー・原則の還流（このプロジェクト → テンプレート正本）

`.claude/` と `KNOWLEDGE.md` の設計原則は各プロジェクトに vendored コピーされた**写し**で、
正本はテンプレートリポジトリ **`soyaho/claude-dev-template`**。
写しだけ直して終わると、次のプロジェクトは直っていないワークフローで立ち上がる。
**還流はプロジェクト側の修正と同じセッションのうちに行う**
（後でまとめてやると「なぜその改良が必要だったか」の記憶が失われる）。

> 中間テンプレート（例: html-game-template）から立ち上げたプロジェクトは、
> まず**直接の親**に還流する。親テンプレート側で「これは開発一般の学びだ」と
> 判断されたものが、さらに上の正本へ還流される（多段還流）。

## 還流対象

1. `.claude/` のスキル・エージェント定義の改良のうち、このプロジェクト固有でないもの
2. `KNOWLEDGE.md` の「設計原則」に足した**開発一般に通じる学び**
   （プロジェクト固有のバグ台帳の行は還流しない。原則に昇格したものだけ）
3. テンプレートが共通基盤（ハーネス等）を配っている場合は、その基盤への修正

## 手順

### 1. 還流対象の分離
- `git diff` で `.claude/` の変更と KNOWLEDGE.md の原則追記を列挙する。
- **プロジェクト固有の知識（成果物の名前・パス・領域固有の語彙）が混ざっていないか確認**。
  混ざっていたら、その部分は CLAUDE.md の「プロジェクト固有の約束事」側に移して
  テンプレート側は汎用のまま保つ（この分離作業こそが「境界の発見」＝テンプレートの育ち方）。

### 2. 正本リポジトリを同席させる
- `add_repo`（owner: `soyaho`, repo: `claude-dev-template`）でこのセッションに追加し、
  指示に従ってクローンする。
- アクセスが無い・拒否された場合は、ユーザーに GitHub 設定でのアクセス許可を
  依頼して**ここで中断**する（黙って諦めて写しだけ直した状態で終えない）。

### 3. 正本に適用
- `.claude/` の diff を正本の同じパスに適用する。
- 設計原則は正本の `KNOWLEDGE.md`「設計原則」に追記する。
  **出自を添える**（どのプロジェクトの・どんな事例から一般化したか）。

### 4. 正本側の整合を確認
- 正本の CLAUDE.md・README・各スキルの相互参照が壊れていないか読み合わせる。
- 正本が検証を持つ場合はそれを回す（このテンプレート自体はドキュメントのみなので、
  参照整合の読み合わせが検証に相当する）。

### 5. コミット＆プッシュ（正本へ）
- メッセージに「**どのプロジェクトで見つけた、どんな改良か**」を書く。

### 6. 整合の確認
- 還流後、プロジェクト側の `.claude/` と正本の `.claude/` の該当部分が一致していることを
  diff で確認する（プロジェクト側だけ新しい／正本だけ新しい状態を作らない）。
