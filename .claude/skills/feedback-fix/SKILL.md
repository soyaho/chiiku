# フィードバックの処理

指摘は「説明・注記を足す」のではなく、**根本原因を診断して構造で直す**。

## 1. 分類（まず原因のクラスを見極める）

指摘を次のどれかに当てる（複数可）。`KNOWLEDGE.md` の対応原則を必ず参照：

| 症状の型 | よくある根本原因 | 参照原則 |
|---|---|---|
| 動かない・進めない・データが消えた | 状態確定の遅延・中断で壊れる永続データ・入力対象の曖昧さ | 原則 1, 6, 7 |
| 環境によって壊れる | 環境マトリクスの穴（未検証の画面サイズ・OS・バージョン） | 原則 4 |
| たまに失敗する（再現しない） | 非同期の競合・遅延実行の前提切れ・固定 sleep | 原則 3, 8 |
| 何をしていいか・どう使うか分からない | 発見可能性の欠如（テストは正解を知っているので検出できない） | 原則 5 |
| 使えるが気持ちよくない・価値を感じない | 摩擦の解消がコアの価値に向かっていない | 原則 0 |

## 2. 診断（再現してから直す）
- 可能なら**利用者の実環境・実寸**で再現する（判読性・使用感は自動検証できない）。
- 「説明・ドキュメント・エラーメッセージを足したくなったら負け」を疑う。
  構造（インターフェースの形・状態の持ち方・因果の見せ方）で解けないか先に考える。
- **フィードバックの翻訳層（原則0）**：指摘を字義通り実装する前に、「この問題を
  **コアの価値の語彙**で解けないか」を必ず一度通す。**ユーザーは摩擦を報告するが、
  価値の欠落は報告しない**——摩擦の解消がコアの価値を削る方向なら立ち止まり、
  代替案を併記してユーザーに提示する。

## 3. 修正
- `/change` パイプラインに乗せる（設計→実装→検証→監査→台帳）。
- 直したら**同種の指摘が再発したら検証が検出できるか**を自問し、
  できないならテスト／invariant を足す。

## 4. 台帳
- `KNOWLEDGE.md` に「症状→根本原因→対策」を1行追記。原則として一般化できるなら
  「設計原則」にも番号を振って足す（→ `/backflow`）。

## 5. 共有
- **何をどう直したか**と**検証結果**を簡潔に報告し、次に見てほしい点を一言添える。
- **定点質問**：次に使ってもらうとき「**◯◯という価値を感じたか？（単に不便が減っただけでなく）**」
  （◯◯＝このプロジェクトのコアの価値。CLAUDE.md 冒頭の一言仕様がそれ）を聞いてもらうよう
  一言添える。摩擦以外を測る唯一の手段。
